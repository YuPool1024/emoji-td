## 0. TL;DR

游戏当前的状态是"可玩闭环完整,但可玩性薄弱"——开局选难度→撑过 10 波→胜/负→退出,这一条**反馈通路**扎实,但**重玩价值/决策深度/新手缓冲/长线动机**几乎为零。本路线图在不扩团队、不重写架构的前提下,用三阶段把游戏从"能玩"推到"想玩":

| 阶段 | 周数(估) | 一句话 | 关键交付 |
|------|---------|--------|---------|
| **P1 顺手级** | 0.5d 一次改 | "我怎么没玩下去?" | 悬停射程预览 + 失败复盘 + 英雄复活条 + 塔克制标签 |
| **P2 中局级** | 1–2d/方向 | "中后期开始有节奏" | 特殊波次 + 终极塔升级 + 暂停/倍速 + 新手引导 |
| **P3 重玩级** | 2–3d/方向 | "通关了我明天还来" | 成就 + 每日挑战(种子) + 多英雄 |
| **UI 重构** | 1d,穿插 | "加新东西不崩" | main.js 拆分 + 统一 UI 组件层 + 状态总览 |

> 任何 P+ 任务落地前,先在 `docs/plans/<topic>-spec.md` 写**单方向的设计+任务拆分**(spec→实现→审查),与 AGENTS.md 流程保持一致。

## 1. 设计支柱(Pillars)

整个 Roadmap 必须同时满足这 4 条,任何"看起来很酷但破坏支柱"的设计点要砍掉。

### Pillar 1 — "看得见的决策"
玩家点的每一下都应该立刻有视觉/听觉/数字反馈。**不要让玩家靠脑补来判断局面。**

具体含义:
- 选中塔后,光标在地图上,立刻画出射程圈和"打不到的红格"(P1 #1)
- 失败时,玩家要立刻知道"因为某波某怪没拦住",而不是只看到 💥 失败
- 终极塔升级时,屏幕要有"我变强了"的高光时刻

### Pillar 2 — "60 秒上手 / 5 分钟上瘾 / 30 分钟见真章"
- **60s 上手**:首局选难度 → 头 30s 内建好第一座塔 → 击中第一只怪(P1 + P2-引导)
- **5min 上瘾**:波 3–5 节奏上扬,玩家开始主动选塔型(P1 #4 克制标签)
- **30min 见真章**:波 7–10 决策才真正重要,前两档难度保护新手,硬核把决策推到极限(P3 难度阶梯)

### Pillar 3 — "再开一局是值得的"
每个 session 结束后,玩家脑子里应该挂着至少 1 句:
- "这把硬核没上完,我再来"
- "我还没解锁终极炮塔,试试"
- "今天是某个种子,我看官方推荐" (P3)

不靠 daily login 强拉,靠"想看看 xxx 怎样"自然回流。

### Pillar 4 — "拒绝堆砌复杂度"
每加一个系统,问两个问题:
1. 玩家是不是会感到"明显多了一个有意思的决策"?
2. 不加它,游戏是否仍然成立?

**靠这句话砍掉了**:
- ❌ "塔装配技能" → 不在 Pillar 2 主张的新手缓冲内
- ❌ "敌人也有天赋树" → 反 Pillar 1
- ❌ "联机协作" → 留作未来远景,不在三阶段内

## 3. 阶段 P1 — 顺手级(可达性 + 反馈)

> 目标:看一遍代码即可落地。每个改进 5–15min。**今天就要能跑起来。**

### P1.1 悬停射程预览
**目的**:把"决策可见性"做到 60% 的游戏场景。

**Player fantasy**: "我点哪个塔,鼠标划过去能看到它能打多远,打不到哪儿"

**Input**:鼠标在 canvas 内移动 + 选中的塔
**现状**:`renderHoverPreview`(main.js L864–898) 已实现:半透明虚线射程圈 + 绿(可放)/红(不可放)预览格 + canFit 判断。P1.1 **仅需补充**以下两项,勿重写已有逻辑:
**Output**:
- 半透明圆(射程 = `range*CELL`),颜色按塔
- 红格标记:`hitsAir=false` 时,光标所在格若可放置但**该格距飞行怪路径近**,提示"打不到飞行"
- 文字"射程 X 格 | 打地/打飞"

**Edge cases**:
- 选中塔为空 → 不画
- 鼠标出 canvas → 隐藏
- 鼠标指在障碍格(grid=0)/已占格(9)/起终点 → 提示"此处不能建塔"

### P1.2 失败复盘面板
**目的**:失败不是终点,是"我下一局该怎么改"的引导。

**Output**(在 `#overlay` 失败界面替换):
```
💥 失败 — 第 7 波
├ 击杀:142 · 漏怪:8 · 最高漏怪波:W7(6个)
├ 塔系偏好:箭×5 电×2 冰×1  (说明: 缺电塔 → 飞行拦不住)
└ [再来一局] [换难度] [复制种子]
```

**注**:种子概念见 P3.3。这里先在内存里记,UI 先放按钮不接逻辑也行。

> **Prereq**: 当前 `createGame`(game.js L8–18) 无 kills/leaks/leaksPerWave/towerBuildHistory 字段。需先在 `GameState` 加埋点:在 `onKill` 计数 + 在 main.js 漏怪处(`baseHp--` 路径)记录 leaksPerWave + 记录建塔历史。

### P1.3 英雄复活倒计时
**目的**:英雄死了玩家就有"立刻花金币 vs 等"的明确选择。

**Mechanic**:
- 英雄 `alive=false` 时,显示一个外环（进度条 0→1 走 60s）
- 倒计时归零自动免费复活(HP=`maxHp*0.5`),或玩家手动金币复活(立即,花费 `h.reviveCost`,同样半血)
- 玩家手动复活时, 进度条立即清零

**数值 placeholder**:
- 倒计时 60s [PLACEHOLDER]
- 自动复活 HP=`maxHp*0.5` [PLACEHOLDER]

> **设计决策（已拍板）**: 手动复活与自动复活统一为半血（`hp = maxHp * 0.5`），避免玩家干等 60s 白嫖。手动复活的优势在于"立即恢复战斗力"，而非"血量更多"。

### P1.4 塔克制标签
**目的**:让玩家在塔栏一眼看到"这个塔打谁"。

**UI**:
```
🏹 箭塔   50💰  [🛬❌][🛡️❌]   ← 红×表示此塔克制不到
⚡ 电塔   90💰  [🛬✓][🛡️✓]
🎯 狙塔  120💰  [🛬✓][🛡️❌]
```

**实现**:直接沿用 `TOWER_TYPES` 已有字段推导,无需新加字段:
- air 克制 = `t.hitsAir` (已存在 bool)
- 破甲克制 = `t.splash > 0` (splash 绕甲,CODEBUDDY.md 明确 armor 只减单体伤害)
渲染时按推导结果画 ✓/❌ 标签即可。

## 4. 阶段 P2 — 中局级(节奏 + 深度)

### P2.1 特殊波次
**目的**:打破"波 1 → 波 10 完全同构"的疲劳感。

> **Prereq**: 当前代码 `ENEMY_FAMILIES`(enemies.js) 只有 4 家族(swarm/shadow/demon/deep),无 boss/elite 类型。需先在 `ENEMY_FAMILIES` 加 `boss`/`elite` 家族,并在 `spawnWave` 中支持"配方调度"(按波号→配方→生成列表)。

**Wave recipe**:
| 波 | 配方 ID | 内容 | 玩家体验目标 |
|----|---------|------|-------------|
| 1–3 | `swarm` | 默认 | 入坑期, 让玩家熟悉 |
| 4 | `swarm_reinforce` | 偶发 elite + 大量小怪 | "突然爆一波", 逼迫升级 |
| 5 | `elite_rush` | 全精英 + 1 boss 提前露头 | "精英海来了" |
| 6–8 | `standard` | 恢复节奏 | 恢复呼吸 |
| 9 | `armor_break` | 重甲为主, 提示玩家需要火/电/炮 | "我得换打法" |
| 10 | `finale` | Boss + 双精英 + 大波 | 收官爆点,胜利画面配合高光 |

**Hook**:在每波开始横幅显示配方名(如 "精英冲锋 — 准备好减速"),给玩家预读 2 秒。

### P2.2 终极塔升级(Lv3 → Tier-3 进化)
**目的**:让"我能再看一会儿"。

> **Prereq**: 当前 `upgradeTower`(towers.js L25) 无等级上限,可无限升级。需先: (1) 加 `level` cap(≤3)使 tier-3 成为终态; (2) 定 tier-3 升级费用公式(当前 `upgradeCost=cost*0.8*level`,L2→3=2.4× 基础价,非"3 倍",需在 spec 中统一)。
> 视觉与弹出确认对话框的 UI 依赖 §6 重构(若未完成则先在 main.js 写临时代码)。

**Tier-3 perk 一览**:
| 塔 | Lv3 触发 | 终极效果 |
|----|---------|---------|
| 🏹 箭塔 | 花费 3 倍基础价 | 三连射(命中 / 命中 / 命中) |
| ⚡ 电塔 | 同 | 链式闪电(命中后弹射 3 个相邻怪) |
| 🎯 狙塔 | 同 | 穿透(一发贯穿一条直线) |
| 🔥 火塔 | 同 | 真 DoT(4s 持续灼烧 ×每秒 X dmg) |
| ❄️ 冰塔 | 同 | 群冻(目标及 1.5 半径内冻 1.5s) |
| 💣 炮塔 | 同 | 散射(3 发扇形霰弹) |

**实现路径**:
1. `TOWER_TYPES.<type>` 加 `tier3: {cost, ...perk}` 
2. `upgradeTower` 检测 `level===2 → 3` 时,弹确认对话框("💎 终极升级 — 花费 250💰,将获得 [perk 名]")
3. 升级后 `tw.perk = type.tier3.perk`, updateTowers 按 type 分支处理
4. 视觉:塔 emoji 套光圈金边

### P2.3 暂停 + 倍速
**目的**:掌控节奏,缓解"波 1 等太久/波 10 太紧张"。

**Mechanic**:
- `Space` — 暂停/继续(toggle)
- `1/2/3` — 1× / 1.5× / 2× 倍速
- 倍速设置影响 `dt`, 物理/AI 一致 scale
- 暂停时所有动画暂停但 HUD 可点击(不阻塞 UI 状态切换)

**Edge cases**:
- 倍速生效时 wave/countdown 也要按比例走完
- 暂停时点塔/出售/升级应仍可用

> **实现陷阱**: 主循环 `loop`(main.js L1127) 有 `dt = Math.min(0.05, ...)` 上限。若 2× 时简单 `dt *= 2` 会被 clamp 吃掉 → 速度不变。正确做法:按倍速 N 在一帧内调用 N 次 `update(dt)`(dt 保持 ≤0.05),或在 clamp **之前**乘倍速并放宽 clamp 上限。spec 必须写明此细节。

### P2.4 首局引导卡(3 张轮播)
**目的**:把"我会玩"的时间从失败后学习变成首局自动教学。

**Cards**:
1. "👋 选塔 → 点格子放塔(至少留一条路给敌人走)"
2. "🦸 选塔时直接点空地 → 部署英雄(范围持续攻击+定身)"
3. "🛰️ 切硬核前,记得:电/狙/炮 可打飞行, 重甲需要持续伤害"

**呈现**:开 MENU 选难度后, 进入 PLAYING 前 8s 静默期, 屏幕底部从右向左滑出卡片, 每张 2.5s 自动切换, 第 3 张完成后保留 👁 "我已了解" 按钮。

**A/B 痕迹**:本引导卡不出现在硬核(硬核默认玩家已会)。

> **协调注意（已决定）**: 选方案 (C)：引导卡 3 张走完（7.5s）→ 出现"我已了解"按钮 → 玩家点击后启动 3.5s 倒计时 → 第一波开始。卡片不覆盖倒计时，体验更干净。

### P3.3 每日挑战(种子地图)
**目的**:拉回流的最高 ROI 杠杆。

**Mechanic**:
- 每 UTC 0 点, 生成一个日期种子(`YYYY-MM-DD` → int)
- 地图生成接受 `seed` 参数, 同种子必出同一图
- 菜单加 "🎲 每日挑战 (剩余 8h 32m)" 入口
- 完成/失败记入 `localStorage['td_daily_runs'][{seed, time, wave, killed, leaked}]`
- 玩家可在分享面板看到自己的"今日最高波数"; 后续可考虑排行(本期不在范围内)

**实现提示**:`generateMap(seed)` 引入 mulberry32 PRNG。需注入位置:
- `carvePath`(map.js L20 洗牌方向) —— 直接调 `Math.random`,非经 utils 助手
- `widenRoad`(map.js L113 概率拓宽) —— 同上
- `choice`/`randInt`(utils.js) —— 经 utils 助手,但底层也是 `Math.random`
建议给 `carvePath/widenRoad/choice/randInt` 统一注入 `rng` 参数,避免覆盖全局 `Math.random`(会污染测试)。

### P3.4 多英雄
**目的**:决策深度 + 角色扮演; 让"再来一局"换成"用 xx 英雄再玩一局"。

**初步英雄池**:
| 角色 | Emoji | 特色 | 数值 placeholder |
|------|-------|------|-------------------|
| 战士(原) | 🦸 | 群定身+中等 DPS | 已有 |
| 法师 | 🧙 | 范围真 DoT + 减速 | dot=10/s, radius=2.0, stick=0 |
| 猎人 | 🏹 | 单体高爆发 + 长程 | dps=50, radius=4.0, stick=2, 永久 1.5× 首击 |

**Mechanic**:

- 不同英雄改变最优布局策略(法师远放, 猎人靠前, 战士居中)
- 解锁进度: 默认解锁战士; 法/猎需要成就 `full_comp` / `pacifist` 解锁

**注**:本节范围**只到数据 + 占位 + 选择面板**; AI 行为按当前 updateHero 模式添加。

## 6. UI 重构(穿插在三阶段中)

### 6.1 现状问题
- `main.js` 已经 41KB, 渲染/输入/状态混在一起
- 加新 UI(成就面板/每日挑战/多英雄选角)时, 都在修改 main.js 的尾部; 单文件越长越难 audit
- 现有 `overlay` / `towerbar` / `hud` 各管一摊, 但互不通约

### 6.2 重构方案(分两步)
**Step A — 抽离 UI 组件层(本次重构)**:
```
js/
  audio.js       ← 现有,不变(音频引擎,index.html 中先于 utils 加载)
  ui.js          ← 新增: 通用 UI 工具(toast/panel/modal/button 的工厂)
  panels/        ← 新增:
    menu.js        (主菜单)
    hud.js         (顶部 HUD)
    towerbar.js    (塔栏 + 射程/克制约定)
    popup.js       (升级/出售/复活弹窗)
    end.js         (胜负面板 + 复盘)
    toast.js       (flash/提示)
  main.js        ← 只保留主循环 / 输入分派 + SFX 初始化
```

**约定**:
- 每个 panel 暴露 `mount(parentEl) → instance`, `update(state)`, `unmount()`
- main.js 只持有"当前选中的 panel 实例 + 1 个 `uiDispatch(action, payload)`"

**Step B — 状态总线**:
- 引入 `store.js`(50 行), 内置发布订阅
- main.js 把 `game` state 改成 store 单例, 各 panel 订阅自己关心的 slice
- 这样成就/每日挑战/多英雄这种"跨局"信息就有个集中地

### 6.3 UI 拆分的代价
- 主线代码 review 时间增加 1–2h(各 panel 单独审)
- 但加新 P+/P++ 方向不再需要 review 一整个 main.js
- 对 ROI 来说, 在做 P2.2 终极塔升级之前必须做, 否则改 6 处的 view 全在 main.js

## 7. 测试策略(本 Roadmap 内)

按现有 AGENTS.md 的 spec→实现→审查模式扩展:

| 方向 | 测试层级 |
|------|---------|
| P1 顺手级 | 浏览器手测(每改一个跑一遍 `index.html`) |
| P2 节奏 | sim_playthrough.js 加 `runSeed(seed)` 支持, 跑 100 次硬核看极端 case |
